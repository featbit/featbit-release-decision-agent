## PR 1 â€” Prisma schema additions

### Goal

Lay the foundation: a project-scoped `CustomerEndpointProvider` table, plus
two columns on `ExperimentRun` to record which data source the run uses and
how to reach it. No behaviour wired up â€” every existing read path still goes
through `track-client.ts`.

### Decisions taken

1. **Plaintext signing secret in v1.** Stored as `String`, not encrypted.
   Phase-2 hardening = either env-key AES at the application layer or move to
   the existing `Vault` model (which today references Azure Key Vault entries,
   not literal secrets). Conscious trade â€” ship plain â†’ harden â†’ migrate is
   cheaper than designing the secret store before the feature has users.
2. **One JSON column for endpoint config**, not multiple typed columns.
   Matches `ExperimentRun.inputData` / `audienceFilters`, which are also
   stringified JSON. Pros: no schema change between Mode A / Mode B / future
   modes. Cons: can't filter on endpoint config in SQL â€” but we never need to.
3. **`projectKey: String`, not FK to a `Project` table.** There is no local
   `Project` model â€” `featbitProjectKey` everywhere is a string identifier
   from the upstream FeatBit backend. New table follows the same pattern.
4. **`dataSourceMode` is on `ExperimentRun`, not `Experiment`.** Same reason
   the metric type/agg fields live on the run: the analyse route reads from
   the latest run, and the data source decision is per-run, not per-experiment
   (a future run of the same experiment can switch warehouses).

### Files changed

- `prisma/schema.prisma` â€” added `CustomerEndpointProvider` model after the
  `Vault` block; added `dataSourceMode` (default `"featbit-managed"`) and
  `customerEndpointConfig` columns to `ExperimentRun`.
- `src/generated/prisma/*` â€” regenerated by `npx prisma generate` (89 ms).
  Type checked clean with `npx tsc --noEmit`; no existing code paths break
  because both new `ExperimentRun` columns are nullable with defaults.

### Migration â€” applied to prod 2026-05-02

Path B was used (direct `npx prisma migrate deploy` against Azure PG, no
throwaway DB verification first). The SQL was reviewed line-by-line against
the user's eyeball check before apply. `prisma migrate status` reports
"Database schema is up to date" â€” `_prisma_migrations` row was written
correctly. No `lock_timeout` fired; in-flight experiments unaffected.

If you need to roll back, see the "Rollback" section at the end.

### Migration file

> **Critical:** never run `npx prisma migrate dev` against the prod Azure PG.
> `migrate dev` is a development command â€” on schema drift it can prompt to
> drop and recreate the database, and it runs `seed.ts` after applying. For
> prod, only ever use `npx prisma migrate deploy`, which only applies
> pending migration files, never drops, never seeds.

The SQL has been hand-written (not generated by `migrate dev`, since we have
no local DB to introspect against without touching prod) and placed at:

```
prisma/migrations/20260502000000_add_customer_endpoint_provider/migration.sql
```

Format matches Prisma's `migrate dev` output exactly (constraint and index
names follow the `<table>_<col>_<kind>` convention, verified against
`20260427000000_add_auth_session/migration.sql`). This means a future
`prisma migrate deploy` will recognise it as a normal Prisma-managed
migration and write the proper row into `_prisma_migrations`.

**Production impact** (Postgres 11+, applies to Azure PG Flexible Server):

- `CREATE TABLE customer_endpoint_provider` â€” new empty table, zero impact.
- `ALTER TABLE experiment_run ADD COLUMN data_source_mode TEXT DEFAULT 'featbit-managed'`
  â€” **metadata-only** since PG 11 (`atthasmissing` / `attmissingval`). No
  table rewrite. The catalog flip takes an `AccessExclusiveLock` on
  `experiment_run` for milliseconds.
- `ALTER TABLE experiment_run ADD COLUMN customer_endpoint_config TEXT`
  (nullable, no default) â€” also metadata-only.

Existing experiments are unaffected: every existing `experiment_run` row
reads `data_source_mode = 'featbit-managed'` virtually, and the analyse
route's existing behaviour stays on the `"featbit-managed"` branch in PR 5.

The only realistic risk is **lock contention**: if a long-running query
holds any lock on `experiment_run` when ALTER runs, the ALTER queues â€” and
all subsequent queries queue behind it (Postgres lock queue is FIFO). The
migration sets `lock_timeout = '5s'` to abort cleanly rather than blocking
indefinitely. If 5s isn't enough during the chosen window, raise it.

**App-code compatibility**: the currently-deployed prod app reads/writes
`experiment_run` via Prisma client. Prisma tolerates DB columns that the
schema doesn't know about (it just doesn't return them) â€” so the migration
can land **before or after** code deploy with no breakage either way.

### How to apply

There are two safe paths depending on whether you want to verify on a
throwaway DB first.

**Path A â€” verify in throwaway Docker Postgres (recommended)**:

```pwsh
# Start temp Postgres
docker run --rm -d --name pg-verify -e POSTGRES_PASSWORD=test -p 5433:5432 postgres:15

# Apply the migration against it (set DATABASE_URL inline, don't overwrite .env)
$env:DATABASE_URL = "postgresql://postgres:test@localhost:5433/postgres"
npx prisma migrate deploy

# Inspect â€” confirm the table and columns landed
npx prisma studio   # opens browser; close when done

# Tear down
docker stop pg-verify
Remove-Item Env:DATABASE_URL    # so Prisma stops pointing at the temp DB
```

If `prisma migrate deploy` succeeds against the throwaway, the same SQL on
prod will succeed too (modulo lock contention).

**Path B â€” apply directly to prod**:

```pwsh
# DATABASE_URL must already be set to the Azure prod URL.
npx prisma migrate deploy
```

`migrate deploy` wraps the SQL in a transaction; if anything fails (bad SQL,
lock_timeout fires, network blip), the transaction rolls back and the DB
stays untouched. Re-running is safe â€” `_prisma_migrations` tracks what's
been applied.

### Rollback

If something does go wrong after apply, the rollback SQL is:

```sql
DROP TABLE "customer_endpoint_provider";
ALTER TABLE "experiment_run" DROP COLUMN "data_source_mode";
ALTER TABLE "experiment_run" DROP COLUMN "customer_endpoint_config";
DELETE FROM "_prisma_migrations" WHERE migration_name = '20260502000000_add_customer_endpoint_provider';
```

Same caveat as the forward migration: ALTER takes AccessExclusiveLock briefly.

### Fan-out / consistency notes for future PRs

- `dataSourceMode` and `customerEndpointConfig` follow the same fan-out rule
  as `primaryMetricEvent` etc.: writes that originate at the Experiment level
  (e.g. expert setup wizard) must propagate to the latest `ExperimentRun` via
  `propagateMetricsToLatestRun()` or an equivalent helper.
- When we add an experiment-level setting that mirrors `dataSourceMode`,
  extend `propagateMetricsToLatestRun()` to copy it; alternatively keep the
  setting run-only and write directly inside the wizard's transaction (the
  pattern `saveExpertSetupAction` already uses).

### What to verify after merge

- `npx prisma generate` succeeds with no warnings.
- `prisma studio` shows the new table empty and the new columns present.
- No existing read path crashes â€” `dataSourceMode` is nullable with a default
  of `"featbit-managed"` so legacy rows degrade gracefully.

## PR 2 â€” Provider CRUD API

### Goal

Stand up the HTTP surface that PR 3's UI will hit. Pure CRUD over
`CustomerEndpointProvider` rows, plus secret rotation. The "test endpoint"
button (Â§8 of the spec) is deferred to PR 4 because it needs the HMAC client.

### Routes

| Verb     | Path                                                                  | Body / response                                          |
|----------|-----------------------------------------------------------------------|----------------------------------------------------------|
| `GET`    | `/api/projects/[projectKey]/customer-endpoints`                       | List, secrets masked.                                    |
| `POST`   | `/api/projects/[projectKey]/customer-endpoints`                       | `{name, baseUrl, timeoutMs?}` â†’ 201 with full secret in `signingSecretPlaintext` (returned ONCE). |
| `GET`    | `/api/projects/[projectKey]/customer-endpoints/[id]`                  | Single, secrets masked. 404 if not found.                |
| `PATCH`  | `/api/projects/[projectKey]/customer-endpoints/[id]`                  | `{action: "update" \| "rotate-secret" \| "clear-secondary", ...}`. Rotate returns full secret ONCE. |
| `DELETE` | `/api/projects/[projectKey]/customer-endpoints/[id]`                  | 204. Referential check on `ExperimentRun.customerEndpointConfig` is deferred to PR 5 â€” JSON column makes the query awkward and the analyser must handle "endpoint disappeared" gracefully anyway. |

### Decisions taken

1. **Server generates `signingSecret`, not the client.** Higher entropy than
   anything an operator would type, and removes a footgun. Format:
   `fbsk_<base64url(crypto.randomBytes(32))>` â€” the `fbsk_` prefix makes the
   token kind obvious in customer logs and config files.

2. **Mask everywhere except create / rotate.** List and get responses include
   `signingSecretMasked: "fbsk_â€˘â€˘â€˘â€˘<last4>"` plus a `hasSecondarySecret`
   boolean. The plaintext only appears in the create / rotate response and is
   never echoed again. UI must surface it once with a "copy and confirm"
   workflow.

3. **Rotation = swap, not just append.** `PATCH {action: "rotate-secret"}`
   moves the current `signingSecret` into `secondarySecret` and writes a new
   primary. A second rotation drops the old secondary. Customers may accept
   either secret during the grace window; FeatBit always signs with the
   primary. `PATCH {action: "clear-secondary"}` ends the grace window.

4. **HTTPS-only baseUrl validation in PR 2.** Full SSRF protection
   (rejecting private/loopback/metadata-service IPs) is deferred to PR 4
   where the actual outbound `fetch()` happens. Two layers of defence is
   correct, but the authoritative check belongs at the call site.

5. **Auth pattern matches existing project-scoped routes.** No
   `requireSession()` call â€” `projectKey` is trusted from the URL, same as
   `api/memory/project/[projectKey]`. Authz hardening across all
   project-scoped APIs is its own cross-cutting concern; not solving it
   inside this feature.

### Files changed

- `src/lib/customer-endpoint-providers.ts` (new) â€” data access + secret
  generation + masking + DTO conversion.
- `src/app/api/projects/[projectKey]/customer-endpoints/route.ts` (new) â€”
  GET (list) and POST (create).
- `src/app/api/projects/[projectKey]/customer-endpoints/[id]/route.ts` (new)
  â€” GET, PATCH (with `action` discriminator), DELETE.
