## PR 1 ‚Äî Prisma schema additions

### Goal

Lay the foundation: a project-scoped `CustomerEndpointProvider` table, plus
two columns on `ExperimentRun` to record which data source the run uses and
how to reach it. No behaviour wired up ‚Äî every existing read path still goes
through `track-client.ts`.

### Decisions taken

1. **Plaintext signing secret in v1.** Stored as `String`, not encrypted.
   Phase-2 hardening = either env-key AES at the application layer or move to
   the existing `Vault` model (which today references Azure Key Vault entries,
   not literal secrets). Conscious trade ‚Äî ship plain ‚Üí harden ‚Üí migrate is
   cheaper than designing the secret store before the feature has users.
2. **One JSON column for endpoint config**, not multiple typed columns.
   Matches `ExperimentRun.inputData` / `audienceFilters`, which are also
   stringified JSON. Pros: no schema change between Mode A / Mode B / future
   modes. Cons: can't filter on endpoint config in SQL ‚Äî but we never need to.
3. **`projectKey: String`, not FK to a `Project` table.** There is no local
   `Project` model ‚Äî `featbitProjectKey` everywhere is a string identifier
   from the upstream FeatBit backend. New table follows the same pattern.
4. **`dataSourceMode` is on `ExperimentRun`, not `Experiment`.** Same reason
   the metric type/agg fields live on the run: the analyse route reads from
   the latest run, and the data source decision is per-run, not per-experiment
   (a future run of the same experiment can switch warehouses).

### Files changed

- `prisma/schema.prisma` ‚Äî added `CustomerEndpointProvider` model after the
  `Vault` block; added `dataSourceMode` (default `"featbit-managed"`) and
  `customerEndpointConfig` columns to `ExperimentRun`.
- `src/generated/prisma/*` ‚Äî regenerated by `npx prisma generate` (89 ms).
  Type checked clean with `npx tsc --noEmit`; no existing code paths break
  because both new `ExperimentRun` columns are nullable with defaults.

### Migration ‚Äî applied to prod 2026-05-02

Path B was used (direct `npx prisma migrate deploy` against Azure PG, no
throwaway DB verification first). The SQL was reviewed line-by-line against
the user's eyeball check before apply. `prisma migrate status` reports
"Database schema is up to date" ‚Äî `_prisma_migrations` row was written
correctly. No `lock_timeout` fired; in-flight experiments unaffected.

If you need to roll back, see the "Rollback" section at the end.

### Migration file

> **Critical:** never run `npx prisma migrate dev` against the prod Azure PG.
> `migrate dev` is a development command ‚Äî on schema drift it can prompt to
> drop and recreate the database, and it runs `seed.ts` after applying. For
> prod, only ever use `npx prisma migrate deploy`, which only applies
> pending migration files, never drops, never seeds.

The SQL has been hand-written (not generated by `migrate dev`, since we have
no local DB to introspect against without touching prod) and placed at:

```
prisma/migrations/20260502000000_add_customer_endpoint_provider/migration.sql
```

Format matches Prisma's `migrate dev` output exactly (constraint and index
names follow the `<table>_<col>_<kind>` convention, verified against
`20260427000000_add_auth_session/migration.sql`). This means a future
`prisma migrate deploy` will recognise it as a normal Prisma-managed
migration and write the proper row into `_prisma_migrations`.

**Production impact** (Postgres 11+, applies to Azure PG Flexible Server):

- `CREATE TABLE customer_endpoint_provider` ‚Äî new empty table, zero impact.
- `ALTER TABLE experiment_run ADD COLUMN data_source_mode TEXT DEFAULT 'featbit-managed'`
  ‚Äî **metadata-only** since PG 11 (`atthasmissing` / `attmissingval`). No
  table rewrite. The catalog flip takes an `AccessExclusiveLock` on
  `experiment_run` for milliseconds.
- `ALTER TABLE experiment_run ADD COLUMN customer_endpoint_config TEXT`
  (nullable, no default) ‚Äî also metadata-only.

Existing experiments are unaffected: every existing `experiment_run` row
reads `data_source_mode = 'featbit-managed'` virtually, and the analyse
route's existing behaviour stays on the `"featbit-managed"` branch in PR 5.

The only realistic risk is **lock contention**: if a long-running query
holds any lock on `experiment_run` when ALTER runs, the ALTER queues ‚Äî and
all subsequent queries queue behind it (Postgres lock queue is FIFO). The
migration sets `lock_timeout = '5s'` to abort cleanly rather than blocking
indefinitely. If 5s isn't enough during the chosen window, raise it.

**App-code compatibility**: the currently-deployed prod app reads/writes
`experiment_run` via Prisma client. Prisma tolerates DB columns that the
schema doesn't know about (it just doesn't return them) ‚Äî so the migration
can land **before or after** code deploy with no breakage either way.

### How to apply

There are two safe paths depending on whether you want to verify on a
throwaway DB first.

**Path A ‚Äî verify in throwaway Docker Postgres (recommended)**:

```pwsh
# Start temp Postgres
docker run --rm -d --name pg-verify -e POSTGRES_PASSWORD=test -p 5433:5432 postgres:15

# Apply the migration against it (set DATABASE_URL inline, don't overwrite .env)
$env:DATABASE_URL = "postgresql://postgres:test@localhost:5433/postgres"
npx prisma migrate deploy

# Inspect ‚Äî confirm the table and columns landed
npx prisma studio   # opens browser; close when done

# Tear down
docker stop pg-verify
Remove-Item Env:DATABASE_URL    # so Prisma stops pointing at the temp DB
```

If `prisma migrate deploy` succeeds against the throwaway, the same SQL on
prod will succeed too (modulo lock contention).

**Path B ‚Äî apply directly to prod**:

```pwsh
# DATABASE_URL must already be set to the Azure prod URL.
npx prisma migrate deploy
```

`migrate deploy` wraps the SQL in a transaction; if anything fails (bad SQL,
lock_timeout fires, network blip), the transaction rolls back and the DB
stays untouched. Re-running is safe ‚Äî `_prisma_migrations` tracks what's
been applied.

### Rollback

If something does go wrong after apply, the rollback SQL is:

```sql
DROP TABLE "customer_endpoint_provider";
ALTER TABLE "experiment_run" DROP COLUMN "data_source_mode";
ALTER TABLE "experiment_run" DROP COLUMN "customer_endpoint_config";
DELETE FROM "_prisma_migrations" WHERE migration_name = '20260502000000_add_customer_endpoint_provider';
```

Same caveat as the forward migration: ALTER takes AccessExclusiveLock briefly.

### Fan-out / consistency notes for future PRs

- `dataSourceMode` and `customerEndpointConfig` follow the same fan-out rule
  as `primaryMetricEvent` etc.: writes that originate at the Experiment level
  (e.g. expert setup wizard) must propagate to the latest `ExperimentRun` via
  `propagateMetricsToLatestRun()` or an equivalent helper.
- When we add an experiment-level setting that mirrors `dataSourceMode`,
  extend `propagateMetricsToLatestRun()` to copy it; alternatively keep the
  setting run-only and write directly inside the wizard's transaction (the
  pattern `saveExpertSetupAction` already uses).

### What to verify after merge

- `npx prisma generate` succeeds with no warnings.
- `prisma studio` shows the new table empty and the new columns present.
- No existing read path crashes ‚Äî `dataSourceMode` is nullable with a default
  of `"featbit-managed"` so legacy rows degrade gracefully.

## PR 2 ‚Äî Provider CRUD API

### Goal

Stand up the HTTP surface that PR 3's UI will hit. Pure CRUD over
`CustomerEndpointProvider` rows, plus secret rotation. The "test endpoint"
button (¬ß8 of the spec) is deferred to PR 4 because it needs the HMAC client.

### Routes

| Verb     | Path                                                                  | Body / response                                          |
|----------|-----------------------------------------------------------------------|----------------------------------------------------------|
| `GET`    | `/api/projects/[projectKey]/customer-endpoints`                       | List, secrets masked.                                    |
| `POST`   | `/api/projects/[projectKey]/customer-endpoints`                       | `{name, baseUrl, timeoutMs?}` ‚Üí 201 with full secret in `signingSecretPlaintext` (returned ONCE). |
| `GET`    | `/api/projects/[projectKey]/customer-endpoints/[id]`                  | Single, secrets masked. 404 if not found.                |
| `PATCH`  | `/api/projects/[projectKey]/customer-endpoints/[id]`                  | `{action: "update" \| "rotate-secret" \| "clear-secondary", ...}`. Rotate returns full secret ONCE. |
| `DELETE` | `/api/projects/[projectKey]/customer-endpoints/[id]`                  | 204. Referential check on `ExperimentRun.customerEndpointConfig` is deferred to PR 5 ‚Äî JSON column makes the query awkward and the analyser must handle "endpoint disappeared" gracefully anyway. |

### Decisions taken

1. **Server generates `signingSecret`, not the client.** Higher entropy than
   anything an operator would type, and removes a footgun. Format:
   `fbsk_<base64url(crypto.randomBytes(32))>` ‚Äî the `fbsk_` prefix makes the
   token kind obvious in customer logs and config files.

2. **Mask everywhere except create / rotate.** List and get responses include
   `signingSecretMasked: "fbsk_‚Ä¢‚Ä¢‚Ä¢‚Ä¢<last4>"` plus a `hasSecondarySecret`
   boolean. The plaintext only appears in the create / rotate response and is
   never echoed again. UI must surface it once with a "copy and confirm"
   workflow.

3. **Rotation = swap, not just append.** `PATCH {action: "rotate-secret"}`
   moves the current `signingSecret` into `secondarySecret` and writes a new
   primary. A second rotation drops the old secondary. Customers may accept
   either secret during the grace window; FeatBit always signs with the
   primary. `PATCH {action: "clear-secondary"}` ends the grace window.

4. **HTTPS-only baseUrl validation in PR 2.** Full SSRF protection
   (rejecting private/loopback/metadata-service IPs) is deferred to PR 4
   where the actual outbound `fetch()` happens. Two layers of defence is
   correct, but the authoritative check belongs at the call site.

5. **Auth pattern matches existing project-scoped routes.** No
   `requireSession()` call ‚Äî `projectKey` is trusted from the URL, same as
   `api/memory/project/[projectKey]`. Authz hardening across all
   project-scoped APIs is its own cross-cutting concern; not solving it
   inside this feature.

### Files changed

- `src/lib/customer-endpoint-providers.ts` (new) ‚Äî data access + secret
  generation + masking + DTO conversion.
- `src/app/api/projects/[projectKey]/customer-endpoints/route.ts` (new) ‚Äî
  GET (list) and POST (create).
- `src/app/api/projects/[projectKey]/customer-endpoints/[id]/route.ts` (new)
  ‚Äî GET, PATCH (with `action` discriminator), DELETE.

### Verification

`npx tsc --noEmit` clean. End-to-end smoke test via `curl` against
`localhost:3000` with prod DB backing ‚Äî created a temp provider under a
unique projectKey, exercised every verb + every error branch, deleted at the
end. All 9 cases passed:

| #  | Case                                | Result          |
|----|-------------------------------------|-----------------|
| 1  | GET empty list                      | `[]`, 200       |
| 2  | POST create                         | 201, plaintext + masked both present |
| 3  | GET list                            | masked only     |
| 4  | GET single                          | masked, 200     |
| 5  | PATCH rotate-secret                 | new plaintext, `hasSecondarySecret: true` |
| 6  | PATCH baseUrl=`http://...`          | 400 with clear msg |
| 7  | POST duplicate name                 | 409             |
| 8  | DELETE                              | 204             |
| 9  | GET after DELETE                    | 404             |

No persistent DB pollution: the temp row was deleted in step 8. Confirmed
via `prisma migrate status` still clean.

## PR 3 ‚Äî Data Warehouse page UI

### Goal

Make the CRUD surface from PR 2 reachable. Match the user's mock from the
original brief: a third top-level card next to "FeatBit Managed" and
"Request a warehouse", plus a full list/manage section beneath.

### Layout

Page top: three cards side-by-side (one row on large screens, stacking on
narrow):

1. **FeatBit Managed Data Warehouse** ‚Äî unchanged, "Connected" badge.
2. **Customer Managed Data Endpoints** ‚Äî new, anchor link to
   `#customer-endpoints` further down the page. Uses `Cable` icon to
   distinguish visually from the `Database` icon of the managed warehouse.
3. **Request a data warehouse** ‚Äî unchanged.

Below the cards, a section component (`CustomerEndpointsSection`) handles
the entire provider lifecycle: list, empty state, add, edit, rotate
secret, clear secondary, delete. The "View schema spec" affordance is
deferred to PR 7 (no live page to link to yet).

### Decisions taken

1. **Single-page, all inline.** No sub-route for the provider list.
   Customers will have at most a handful of providers, sub-pages would
   add navigation friction for no clarity gain. PR 7's docs page is
   the only sub-route under `/data-warehouse/`.

2. **Page redesigned as a unified list with type filters.** The original
   3-cards-on-top + section-below layout was abandoned after iterating
   through several broken UX shapes (clickable anchor card that felt
   inert; passive status panel that felt static; expand-on-click toggle
   that hid useful info by default). Final shape: a single list of
   data-source rows of three types, with a type-filter chip row above
   it and a single "Request a data warehouse" button on the right. Row
   types:
     - **FeatBit Managed** (always 1 row) ‚Äî clicks open `/data/apis-sdks`
       in a new tab, since SDK setup is the actual onboarding flow for
       the managed warehouse.
     - **Customer Managed Data Endpoint** (0..N rows) ‚Äî clicks open the
       edit dialog (PR 3 form, unchanged).
     - **External Data Warehouse** ‚Äî *not stored as rows*. Requesting a
       managed connector stays fire-and-forget email; the existing
       `RequestProviderDialog` is now opened via the chooser instead of
       its own card. No `External` filter chip, since there are no
       External rows.
   The "Request a data warehouse" button opens an `AddDataSourceChooserDialog`
   with two options: pick "Customer Managed Endpoint" ‚Üí opens the add
   form; pick "External Data Warehouse" ‚Üí opens the email dialog.

3. **Two-step delete confirmation inline.** "Delete" button reveals a
   "Confirm delete" button in the same dialog; no modal-on-modal. Avoids
   the visual noise of a separate confirm dialog while still preventing
   one-click destruction.

4. **Secret-reveal modal blocks until acknowledged.** `SecretRevealDialog`
   suppresses backdrop-click and Escape-to-close until the operator
   ticks "I've saved this secret somewhere I can retrieve it later."
   The full secret is shown once and never persisted in component state
   past `onClose()`.

5. **Auth + project context via existing `useAuth()`.** Same pattern as
   `AiMemoryClient` ‚Äî `currentProject?.key` is the projectKey for the
   API URL. Section renders an "select a project" placeholder if there's
   no current project (shouldn't happen inside `(dashboard)` layout,
   defensive only).

### Files changed

- `src/components/data-warehouse/types.ts` (new) ‚Äî client-side mirrors
  of `ProviderPublic` / `ProviderWithSecret` to avoid pulling the
  server-only `@/lib/customer-endpoint-providers` module into client
  bundles.
- `src/components/data-warehouse/data-warehouse-client.tsx` (new) ‚Äî
  unified list view. Owns filter state, fetches Customer Endpoints,
  orchestrates the chooser / add-form / edit-form / secret-reveal
  dialog stack. Hosts the FeatBit-Managed and Customer Endpoint row
  components inline.
- `src/components/data-warehouse/customer-endpoint-form-dialog.tsx`
  (new) ‚Äî handles both add and edit modes via a discriminated `mode`
  prop. Edit mode includes rotate, clear-secondary, and delete-with-
  confirm.
- `src/components/data-warehouse/secret-reveal-dialog.tsx` (new) ‚Äî
  blocking modal for the one-shot plaintext-secret display.
- `src/components/data-warehouse/add-data-source-chooser-dialog.tsx`
  (new) ‚Äî two-option chooser between "Customer Managed Endpoint" and
  "External Data Warehouse" (request a connector).
- `src/components/data-warehouse/request-provider-dialog.tsx`
  (modified) ‚Äî accepts both `trigger` (uncontrolled) and
  `open`/`onOpenChange` (controlled) so the chooser can open it
  programmatically.
- `src/app/(dashboard)/data-warehouse/page.tsx` ‚Äî thin wrapper around
  `<DataWarehouseClient />` plus a header.

### Verification

- `npx tsc --noEmit` clean.
- `next dev` compiles all new files without warnings or errors
  (verified via `.next/dev/logs/next-development.log` after a request
  triggered build of the data-warehouse route ‚Äî no entries beyond the

- `GET /data-warehouse` returns 307 ‚Üí `/login?redirect=/data-warehouse`

**Browser verification still required by the operator** ‚Äî automated
tooling here can't drive the dialog flow. Human checklist:

- [ ] Open `/data-warehouse`. Confirm three cards appear in the top row.

## When fully shipped ‚Äî AGENTS.md additions

When all PRs are merged, the following architectural rules get promoted into
`modules/web/AGENTS.md` (they belong with the existing "Metric Vocabulary &
Fan-out Contract" section since they're cross-cutting):

- The two valid data source paths in the analyse route, and how they branch
  on `dataSourceMode`.
- The same fan-out rule for `dataSourceMode` / `customerEndpointConfig` as
  for the metric columns.
- HMAC signing format (so future API additions stay consistent).
- The stats-shape normalisation rule: customer endpoint may return either
  `{n, mean, stddev}` or `{n, sum, sum_squares}` for continuous; everything
  downstream of the client sees `{n, sum, sum_squares}` (the existing internal
  shape) ‚Äî `metricMoments()` is unchanged.
