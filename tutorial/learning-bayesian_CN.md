> 本教程通过阅读 `experiment-workspace` 中的 `analyze-bayesian.py` 和 `references/analysis-bayesian.md` 逐步学习贝叶斯 A/B 测试的实现逻辑。
> 记录了学习过程中的真实问答，作为持续学习的 context。

---

## 第一章：为什么用 Bayesian？

传统 A/B 测试（频率派）问的是反事实问题：

> "假设 treatment **没有效果**，我观察到这个数据的概率是多少？"

这就是 p-value。它回答的是一个间接问题，不直观。

Bayesian 直接回答：

> "看完这批数据之后，treatment **真的更好**的概率是多少？"

输出就是 `P(win) = 94%`，意思是：基于当前数据，有 94% 的概率 treatment 确实更好。可以直接用于决策。

---

## 第二章：两种指标类型

### 判断标准

看你的原始数据是什么形状：

| 问题 | 指标类型 | 例子 |
|------|---------|------|
| 每个用户只有"发生"或"没发生" | 比例指标（proportion） | 点击了没有、注册了没有、购买了没有 |
| 每个用户有一个具体数值 | 连续指标（continuous） | 花了多少钱、停留了多少秒、看了几篇文章 |

### 输入数据格式

```json
// 比例指标：只需要 n 和 k
"cta_clicked": {
  "control":   {"n": 5000, "k": 300},
  "treatment": {"n": 5050, "k": 350}
}

// 连续指标：需要 n、sum、sum_squares
"revenue_per_user": {
  "control":   {"n": 5000, "sum": 45000.0,  "sum_squares": 820000.0},
  "treatment": {"n": 5050, "sum": 48020.0,  "sum_squares": 901000.0}
}
```

### sum_squares 是什么？

`sum_squares` 是每个用户的值**各自平方后再加总**：

```
sum_squares = x₁² + x₂² + x₃² + ... + xₙ²
```

注意：**不是** `(x₁ + x₂ + ... + xₙ)²`，不是 sum 的平方。

**为什么需要它？** 因为方差的公式是：

```
var = (sum_squares - sum²/n) / (n - 1)
```

只有 `sum` 算不出方差，必须有 `sum_squares`。实际项目里数据在数据库里，直接聚合查询即可：

```sql
SELECT
  COUNT(*)            AS n,
  SUM(value)          AS sum,
  SUM(value * value)  AS sum_squares
FROM events
WHERE variant = 'control'
```

一条 SQL，三个数字，脚本拿到这三个数就能算出均值和方差。

### 边界情况：inverse 指标

错误率、延迟等"越低越好"的指标，加 `"inverse": true`，P(win) 的含义变为"treatment 的值更低的概率"。

## 第三章：所有变量的定义与计算

### 3.1 比例指标案例

**场景：** 测试新版 CTA 按钮是否能提升点击率。

**原始数据：**

| 变体 | 曝光人数 n | 点击人数 k |
|------|-----------|-----------|
| control | 5000 | 300 |
| treatment | 5050 | 350 |

**mean（均值）= 转化率：**

```
mean_a = 300 / 5000 = 0.06     （6.00%）
mean_b = 350 / 5050 = 0.06931  （6.93%）
```

**var（方差）= 单个用户行为的离散程度：**

对比例指标，每个用户非 0 即 1，方差由均值本身决定：

```
var_a = 0.06 × (1 - 0.06)      = 0.0564
var_b = 0.06931 × (1 - 0.06931) = 0.06450
```

均值越接近 0.5，方差越大（结果最不确定）；越接近 0 或 1，方差越小。

**μ_rel（相对效应点估计）= 观测到的相对提升：**

```
μ_rel = (mean_b - mean_a) / mean_a
      = (0.06931 - 0.06) / 0.06
      = +15.52%
```

**se（标准误差）= 对 μ_rel 这个估计有多不确定：**

用 Delta Method 计算：

```
se = sqrt(
    var_b / (n_b × mean_a²)
  + var_a × mean_b² / (n_a × mean_a⁴)
)
≈ 8.79%
```

se 越大 = 样本越少或方差越大 = 估计越不可靠。
分母里有 `mean_a²`，所以 baseline 转化率越低，se 越大——低转化率实验天然更不确定。

---

### 3.2 连续指标案例

**场景：** 测试新结账流程是否提升每用户收入。

**原始数据：**

| 变体 | n | sum | sum_squares |
|------|---|-----|-------------|
| control | 5000 | 45000.0 | 820000.0 |
| treatment | 5050 | 48020.0 | 901000.0 |

**计算过程：**

```
# control
mean_a = 45000 / 5000 = 9.0（元/人）
var_a  = (820000 - 45000²/5000) / (5000-1)
       = (820000 - 810000) / 4999
       ≈ 2.0004

# treatment
mean_b = 48020 / 5050 ≈ 9.509（元/人）
var_b  = (901000 - 48020²/5050) / (5050-1)
       ≈ 88.01
```

注意：连续指标的 var 通常比比例指标大很多（用户间收入差异大），但 n 足够大时 se 仍然可以很小。

```
μ_rel = (9.509 - 9.0) / 9.0 = +5.65%
se    ≈ 1.64%

P(win) = norm.sf(0, 0.0565, 0.01637) ≈ 99.97%
```

## 第四章：后验分布 N(μ_rel, se²)

### N 是什么？

`N` 是**正态分布（Normal / Gaussian Distribution）**。记号 `N(μ, σ²)` 中：

- `μ` = 均值（最可能的值）
- `σ²` = 方差，`σ = se` = 标准差（不确定程度）

注意：`N(μ, σ²)` 的第二个参数是方差（σ²），不是标准差（σ）。代码里操作的是 `se`（即 σ），因为标准差和原始数据单位相同，更直观。

### 为什么是分布而不是一个数？

因为你**不知道真实的 δ 是多少**，只能说它"大概在哪个范围"。

类比：朋友说"我大概 3 点到，但可能早或晚半小时"——给你的不是一个确定时间，而是一个带不确定性的估计。正态分布就是这种不确定性的数学表达：

```
μ = 最可能的值（3点）
σ = 不确定程度（半小时）
```

在 A/B 测试里，观测到的 δ = +15.52% 只是一次采样结果。如果再跑一次实验，数据略有不同，δ 也会略有不同。正态分布描述的是：**真实的 δ 落在各个位置的可能性有多大**。

### 为什么能用正态分布近似？

来自**中心极限定理（CLT）**：

> 当样本量足够大时，样本均值的分布趋近于正态分布——无论原始数据是什么分布。

这也是为什么脚本要求每个变体至少 **30 次转化**——低于这个数，CLT 还没生效，正态近似不可靠，P(win) 和 risk 的值可能是误导性的。

## 第五章：P(win) 的计算

### 直觉

`P(win) = δ > 0 的概率 = 后验曲线中 0 右侧的面积`

```
         ▲
         │         ┌───┐
         │       ┌─┘███└─┐
         │      ─┘███████└─
         │   ───████████████───
         └─────┬───────────────▶ δ（相对提升）
               0   +μ_rel
         ←────→←──────────────→
          P(ctrl好)  P(win)
```

因为均值 +15.52% 在 0 的右边，右侧面积（P(win)）会很大。

### 代码实现

```python
ctw = norm.sf(0.0, loc=μ_rel, scale=se)
```

`sf` = survival function = `1 - CDF`，即**大于某个值的概率**：

```
norm.sf(0, loc=0.1552, scale=0.0879)
= P(δ > 0)，其中 δ ~ N(0.1552, 0.0879²)
= P(treatment 的真实提升 > 0)
= P(win)
≈ 96.1%
```

均值离 0 约 1.76 个标准差（0.1552 / 0.0879），所以概率很高。

## 第六章：这些计算和贝叶斯的关系

很多人会困惑：上面的计算看起来都是"频率派"的工具（均值、方差、标准误差），贝叶斯在哪里？

**贝叶斯定理：**

```
后验 ∝ 先验 × 似然
```

**默认模式（flat prior，`proper: false`）：**

先验是"平坦的"——对所有可能的 δ 值一视同仁，没有任何偏好。这时候：

```
后验 ∝ 1 × 似然 = 似然
```

后验完全等于数据的似然函数。`N(μ_rel, se²)` 是对似然函数的高斯近似——当样本足够大时（CLT），似然函数形状近似正态。

所以整个计算流程是：

```
观测数据
  → 算出点估计 μ_rel 和不确定性 se    （频率派的工具）
  → 用 N(μ_rel, se²) 近似后验         （贝叶斯的框架）
  → 从后验里算 P(win)、CI、risk        （贝叶斯的输出）
```

**贝叶斯的价值在于框架，不在于计算方法。** 它让你能说出"P(win) = 96%"这句话——在频率派里，你不能直接说"treatment 更好的概率是 96%"，只能说"p < 0.05，拒绝零假设"。

**启用先验（`proper: true`）时，贝叶斯才真正"介入"计算：**

```
post_mean = (data_mean/data_var + prior_mean/prior_var)
          / (1/data_var + 1/prior_var)
post_std  = sqrt(1 / (1/data_var + 1/prior_var))
```

先验会把后验均值拉向历史经验。样本小时先验影响大，样本大时数据压制先验——这是纯频率派做不到的。

## 第七章：95% 可信区间（Credible Interval）

### 频率派置信区间 vs Bayesian 可信区间

**频率派置信区间（Confidence Interval）** 的严格含义很反直觉：

> "如果我重复做 100 次实验，有 95 次的置信区间会包含真实值。"

它描述的是**区间这个方法**的长期表现，不是说"真实值有 95% 的概率在这个区间里"。

**Bayesian 可信区间（Credible Interval）** 直接说：

> "真实的 δ (delta) 有 95% 的概率落在这个区间里。"

这才是大多数人以为置信区间在说的事情。

---

### 为什么 Bayesian 能这么说？

关键在于：**Bayesian 把 δ 本身当作一个随机变量，而频率派不这么做。**

**频率派的世界观：**
> 真实的 δ 是一个固定的数，只是你不知道它是多少。固定的数没有概率可言——它要么在区间里，要么不在。所以频率派只能说"我这个区间的构造方法，长期来看有 95% 的命中率"。

**Bayesian 的世界观：**
> 我对真实的 δ 有不确定性，这种不确定性可以用概率来描述。δ 对我来说是随机的——不是因为它真的在变，而是因为**我不知道它的真实值**。因此可以说"给定我观察到的数据，δ 落在这个区间的概率是 95%"。

**日常类比：**

你找不到钥匙：
- 频率派说：钥匙在一个固定的地方，我没法说"钥匙有 70% 的概率在厨房"，我只能说"我的搜索策略长期来看有 95% 的成功率"。
- Bayesian 说：我不知道钥匙在哪，根据我的记忆，钥匙有 70% 的概率在厨房，20% 在卧室，10% 在其他地方。

Bayesian 允许你把**主观不确定性**用概率表达出来。

---

### 脚本里怎么算的？

后验是 `N(μ_rel, se²)`，95% 可信区间就是这条正态曲线**中间 95% 的区域**：

```
ci_lower = μ_rel - 1.96 × se
ci_upper = μ_rel + 1.96 × se
```

为什么是 1.96？正态分布的性质：均值左右各 1.96 个标准差，恰好覆盖 95% 的面积。

```python
z_half   = norm.ppf(1.0 - 0.05 / 2)   # = 1.96
ci_lower = μ_rel - z_half × se
ci_upper = μ_rel + z_half × se
```

用 CTA 点击率案例：

```
μ_rel = +15.52%,  se = 8.79%

ci_lower = 15.52% - 1.96 × 8.79% = +0.29%
ci_upper = 15.52% + 1.96 × 8.79% = +30.75%

95% 可信区间：[+0.29%, +30.75%]
```

意思是：**看完这批数据之后，真实提升有 95% 的概率在 +0.29% 到 +30.75% 之间。**

区间很宽是因为 se = 8.79% 较大——样本量还不够多，随着样本增加 se 变小，区间会收窄。

---

### 可信区间有什么用？

**1. 判断结果有没有实际意义**

P(win) 只告诉你方向，CI 告诉你量级：

```
案例 A：P(win) = 96%，CI = [+0.1%, +1.2%]   → 提升幅度太小，值得发布吗？
案例 B：P(win) = 96%，CI = [+8%,  +25%]    → 最保守估计也有 +8%，有业务价值
```

**P(win) 告诉你方向，CI 告诉你量级。**

**2. 判断实验是否需要继续跑**

```
早期（样本少）：CI = [-5%, +35%]   → 区间很宽，结果还很模糊，继续跑
中期（样本中）：CI = [+3%, +28%]   → 区间收窄，但范围还大
后期（样本足）：CI = [+10%, +21%]  → 区间稳定，结果可信，可以决策
```

**3. 和业务的 MDE 对比**

MDE（Minimum Detectable Effect）= 最小有意义的提升，低于这个值不值得发布。

比如 MDE = +5%：

```
CI = [+0.1%, +12%]  → 下界低于 MDE，继续跑
CI = [+6%,  +18%]   → 整个区间都在 MDE 以上，可以决策
CI = [+0.5%, +3%]   → 整个区间都在 MDE 以下，即使 treatment 更好也不值得发布
```

## 第八章：risk（风险/期望损失）

### 直觉

risk 回答的是两个对称的问题：

| 问题 | 对应的 risk |
|------|------------|
| 如果我**不发布** treatment，但 treatment 其实更好——我损失了多少机会？ | `risk[ctrl]` |
| 如果我**发布了** treatment，但 control 其实更好——我损失了多少？ | `risk[trt]` |

两个值都是"期望损失"——不是最坏情况，而是**按概率加权的平均损失**。

---

### 公式

```
risk[ctrl] = E[max(0,  δ)] = ∫₀^∞  δ · p(δ) dδ
risk[trt]  = E[max(0, -δ)] = ∫₋∞^0 |δ| · p(δ) dδ
```

用人话说：

- `risk[ctrl]`：把后验曲线里 **δ > 0 的部分**，按概率加权平均——如果你留在 control 而 treatment 更好，平均损失多少相对提升
- `risk[trt]`：把后验曲线里 **δ < 0 的部分**，按概率加权平均——如果你发布 treatment 而 control 更好，平均损失多少

---

### 案例 1：比例指标（CTA 点击率）

后验：`δ ~ N(+15.52%, 8.79%²)`，P(win) = 96.1%

因为后验均值在 0 的右边很远：
- 曲线中 δ < 0 的面积只有 3.9% → `risk[trt]` 很小
- 曲线中 δ > 0 的面积有 96.1% → `risk[ctrl]` 较大

实际输出：
```
risk[ctrl] = 0.1498   （不发布的话，期望损失约 15% 的 control 均值）
risk[trt]  = 0.0026   （发布错了的话，期望损失约 0.26% 的 control 均值）
```

**结论：发布的代价极低（0.26%），不发布的代价极高（15%）。应该发布。**

如何理解 0.26%：control 点击率是 6%，`risk[trt] = 0.0026` 意味着平均损失 0.0026 × 6% ≈ 0.016% 的点击率——几乎可以忽略不计。

---

### 案例 2：连续指标（每用户收入）

后验：`δ ~ N(+5.65%, 1.64%²)`，P(win) = 99.97%

后验均值离 0 更远（约 3.4 个标准差），结果更确定：

```
risk[ctrl] = 0.0565   （不发布的话，期望损失约 5.65% 的 control 均值）
risk[trt]  = 0.0000   （发布错了的话，期望损失几乎为 0）
```

**结论：P(win) 极高，risk[trt] 趋近于 0。发布。**

如何理解：control 均值是 9.0 元/人，`risk[trt] ≈ 0` 意味着即使发布错了，损失也微乎其微。

---

### 这和"发布成本"的关系

你发布 treatment 之前，需要判断：**如果我错了，损失能 cover 发布本身的成本吗？**

`risk[trt]` 就是对这个问题的精确量化：

| risk[trt] | 含义 |
|-----------|------|
| < 0.001 | 损失 < 0.1%，可以放心发布 |
| 0.001 – 0.01 | 损失 0.1%–1%，大多数产品决策可以接受 |
| > 0.01 | 损失 > 1%，结合业务背景判断 |

---

### 案例 3："结合业务背景判断"是什么意思？

**场景：** 测试新结账流程，结果如下：

```
P(win)       = 87%
risk[trt]    = 0.018   （损失约 1.8% 的 control 均值）
control 均值  = 每用户收入 50 元
```

`risk[trt] = 0.018` 意味着：如果发布错了，平均损失 0.018 × 50 = **0.9 元/人**。

**同样的数字，两种业务背景，两个不同结论：**

**情境 A：日活 100 万的电商平台**
```
每天损失 = 0.9 元 × 1,000,000 人 = 90 万元/天
```
P(win) 只有 87%，代价太高。→ **继续跑，等 risk[trt] 降到 0.005 以下再决策。**

**情境 B：日活 1000 人的早期产品**
```
每天损失 = 0.9 元 × 1,000 人 = 900 元/天
```
工程改动只需半天，业务上迫切需要验证方向。→ **可以接受，发布并持续监控。**

**核心结论：** `risk[trt]` 给你的是一个**相对比例**，不是绝对金额。你需要自己换算成业务能理解的单位，再对比发布成本和业务压力，才能做出判断。这一步判断属于 `evidence-analysis` 的工作，不是脚本自动给出的。

## 第九章：SRM 检查（Sample Ratio Mismatch）

### 什么是 SRM？

你配置的是 50/50，但实际收到的数据是 control=5000，treatment=4750——这个差异是正常的随机波动，还是出了问题？

SRM 检查用 **χ² (卡方) 检验**来回答这个问题：观测到的流量比例，和配置的预期比例，差异是否超出了随机波动的范围。

```python
srm_p = srm_check([n_control, n_treatment])
# p < 0.01 → 红旗，流量分配有问题
```

p-value < 0.01 时输出：
```
⚠ POSSIBLE IMBALANCE — investigate before interpreting results
```

---

### 为什么 Bayesian 分析里需要 SRM 检查？

Bayesian 分析的前提假设是——control 和 treatment 的用户构成是可比的。

如果流量分配出了问题，比如 treatment 组阴差阳错地吸引了更多高价值用户，那么观测到的 δ 不是因为 treatment 更好，而是因为**两组用户本来就不一样**。此时 P(win) = 96% 是假的——垃圾进，垃圾出。

**类比：** 你想比较两家餐厅的菜好不好，但一家店的顾客全是美食家，另一家全是随便吃的上班族。两组人打分不一样，不代表菜的质量不同——是样本本身就不对。

所以正确的流程是：

```
收到数据
  → 先跑 SRM 检查
  → SRM 通过 → 再跑贝叶斯分析 → 读 P(win) / risk / CI
  → SRM 不通过 → 停下来排查，不要看贝叶斯结果
```

脚本里 `srm_block` 永远出现在 `analysis.md` 的最顶部，顺序是固定的。

---

### 为什么 SRM 检查用 p-value，而不是 Bayesian？

SRM 检查和贝叶斯分析是两个独立的问题，用不同的工具：

| 问题 | 工具 | 为什么 |
|------|------|--------|
| treatment 是否更好？ | Bayesian 后验 | 需要直接的概率表达，用于决策 |
| 流量分配是否正常？ | 频率派 χ² 检验 | 只需要一个"是/否"的警报，p-value 够用 |

SRM 只需要回答一个二元问题："这个偏差是随机波动还是系统性问题？" 对于这种异常检测，p-value 是完全合适的工具。

> 贝叶斯用于**量化效果**，频率派用于**检测异常**。两者在同一个脚本里各司其职，没有冲突。

---

### feature flag 场景特别容易出 SRM 的原因

**1. SDK 初始化时序问题**
用户进入页面，flag 还没评估完，事件已经上报了。部分用户被分配了 variant 但没被正确记录为"已曝光"。

**2. 缓存/持久化不一致**
用户第一次访问被分到 treatment，清了 cookie 或换设备后又被分到 control，导致同一个用户在两个桶里都被计数。

**3. 分阶段 rollout**
没有一次性打开到 50/50，而是先 10%、再 30%、再 50%——实验窗口开始时间和"真正 50/50"的时间对不上，早期数据污染了结果。

**4. Bot 流量**
爬虫被分配了 variant，但行为和真实用户完全不同，且往往集中在某一个桶。

---

### SRM 检查的输入：n 必须是唯一用户数

SRM 检查只需要一个输入：**每个 variant 的曝光人数 `n`**。

脚本从 `input.json` 的 primary metric 里直接读取：

```python
ns = [pm.get(v, {}).get("n", 0) for v in all_variants]
srm_p = srm_check(ns)
```

**关键：`n` 必须是唯一用户曝光数（DISTINCT user），不是事件次数。**

| 数据来源 | 含义 | 用于 SRM？ |
|---------|------|-----------|
| flag evaluation 次数 | 同一个用户可能被评估多次 | ❌ 不对 |
| 唯一用户曝光数 | 每个用户只算一次 | ✅ 正确 |

正确的查询方式：

```sql
SELECT variant, COUNT(DISTINCT user_key) AS n
FROM exposures
WHERE experiment_slug = '<slug>'
  AND timestamp BETWEEN start AND end
GROUP BY variant
```

---

### SRM 检查的局限性

SRM 是**事后验证**，不能保证输入数据的正确性。脚本无法知道你提供的 `n` 是唯一用户数还是事件次数——这个责任在数据收集环节。

所以正确理解 SRM 结果的方式：

- **SRM 通过（p ≥ 0.01）：** 基于你提供的 `n` 值，流量分配看起来正常。前提是 `n` 的定义正确。
- **SRM 不通过（p < 0.01）：** 除了排查流量分配问题，也要检查数据收集方式是否正确——`n` 的定义错误也会导致 SRM 报警。

> SRM 检查的质量上限，取决于 `n` 的定义是否正确。脚本无法验证这一点，只能通过文档契约把责任明确交给数据收集环节。

## 第十章：先验的使用（proper prior）

### 先验是什么？

先验（prior）是你在看到实验数据**之前**，对 δ 的已有信念。

脚本支持两种模式：

```markdown
prior:
  proper: false   # 默认：平坦先验，后验完全由数据决定
  proper: true    # 启用：用历史经验影响后验
  mean:   0.0     # 你预期的提升方向（0 = 没有预期方向）
  stddev: 0.3     # 你认为提升幅度的合理范围（±30%）
```

---

### 什么时候用先验？

**正确的来源：实验开始之前的独立信息**

- 历史上类似实验的结果
- 行业经验或领域知识
- 和本次实验数据完全无关的外部数据

**错误的来源：本次实验的早期数据**

如果你用实验本身的前 3 天数据设置先验，再用包含这 3 天的全部数据算后验——早期数据被用了两次，结果偏向早期噪声，统计上不允许。

---

### "先跑 3-5 天，再开新实验用这个数据做先验"——可以吗？

**完全可以，这是正确的用法。** 关键是两个实验的数据必须完全分开：

```
实验 A：第 1-5 天，收集基线数据
  → 得到 δ ≈ 12%，se ≈ 8%
  → 设置先验：mean = 0.12, stddev = 0.08

实验 B：第 6 天重置观测窗口，开新实验
  → input.json 只包含第 6 天之后的数据
  → 先验 + 新数据 → 后验
```

| 情况 | 合法？ |
|------|--------|
| 实验 A 的数据做先验，实验 B 用全新数据 | ✅ 合法 |
| 前 3 天做先验，用包含这 3 天的全部数据算后验 | ❌ 数据用了两次 |

---

### 先验的实际效果

| 样本量 | prior 的影响 |
|--------|------------|
| 小（n < 200） | 强——结果被拉向 prior mean |
| 中等 | 部分收缩 |
| 大（n > 1000） | 数据压制先验，几乎无影响 |

样本足够大时，用不用先验结果几乎一样。先验只在**样本小、噪声大**的早期阶段有明显影响。
